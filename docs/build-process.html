Executive summary

What was built

The project is a personal resume website that turns a traditional resume into a focused, mobile-friendly web presence. It highlights education, work experience, leadership, skills, contact details, professional profiles, and a working contact form, while also documenting the hosting and deployment work behind the finished page.

Professional first impression

The homepage presents a strong identity section with name, portrait, concise summary, contact card, and links to GitHub and LinkedIn.

Resume content online

Core resume sections were structured as scannable website sections: About Me, Education, Work Experience, Leadership, Skills, and Contact Me.

Deployment-ready foundation

Hugo, Caddy, and the Linode workflow produce optimized static output, stage it safely, and publish it only after the build is verified.

Progression

Build timeline

The Codex sessions show a more complete order of operations: server setup, first deploy, preview workflow, theme exploration, resume-content conversion, portrait/social-link iteration, contact form integration, and final migration back to the local repository.

Phase 1June 17, 2026

Linode, Ubuntu, Hugo, and Caddy plan

The project started with the hosting architecture rather than the page design. The chosen stack was a small Linode running Ubuntu, Hugo for static-site generation, and Caddy for serving the site with automatic HTTPS.

  • Selected a low-cost Linode Nanode-style hosting path.
  • Used Ubuntu as the server operating system.
  • Planned Caddy as the web server because it handles HTTPS cleanly.
  • Outlined SSH key, firewall, DNS, and non-root user setup.

Phase 2June 17, 2026

First Hugo scaffold and safe staging workflow

Once the Linode was reachable, the Hugo source was located at /home/josh/resume-site and the Caddy web root was identified as /usr/share/caddy. The initial Hugo scaffold was not ready to publish, so a minimal first resume shell was created before touching the live site.

  • Configured the real production URL in hugo.toml.
  • Added the first hand-built homepage and custom stylesheet.
  • Built to /home/josh/resume-deploy as a staging directory.
  • Kept the live Caddy directory untouched until validation passed.

Phase 3June 17-18, 2026

First deploy and live verification

The first version was deployed by copying the staged Hugo output into Caddy's protected web root with a sudo command entered by the site owner.

  • Backed up the existing Caddy site before replacing files.
  • Published staged output from /home/josh/resume-deploy.
  • Verified apex and www hostnames over HTTPS.
  • Confirmed fingerprinted CSS and Caddy responses loaded correctly.

Phase 4June 18-19, 2026

Preview workflow and theme exploration

A safe edit-preview-publish loop was established before design experimentation continued. A Hugo resume theme was explored, then preserved on a branch and removed from main after deciding the hand-built homepage was a better fit.

  • Ran hugo server --bind 127.0.0.1 on the Linode.
  • Used an SSH tunnel to preview the remote server at local port 1313.
  • Saved the first deployed site before installing the theme.
  • Created a resume-theme branch and restored the custom HTML version.

Phase 5June 19-20, 2026

Resume content converted into web sections

The resume PDF was used as the source of truth for the site content. The information was reorganized into readable web sections rather than copied as a flat document.

  • Extracted the resume content and mapped it into the homepage.
  • Preserved the existing red visual system and navigation menu.
  • Added education, IT experience, leadership, skills, coursework, and contact details.
  • Verified desktop and mobile readability through the preview server.

Phase 6June 20-22, 2026

Hero, portrait, social links, and contact card

The top of the page was iteratively refined from a text-only resume header into a personal hero with a portrait, profile links, location, phone, and email.

  • Added a circular portrait to the right of the name.
  • Added LinkedIn first, then verified and added the GitHub profile.
  • Replaced text buttons with clickable PNG social logos.
  • Moved phone, email, location, and social icons into a shared contact card.
  • Checked mobile centering, no-overflow behavior, and icon alignment.

Phase 7June 22, 2026

Contact form research and Formspree integration

Because Hugo is static, the form needed a hosted endpoint or a custom backend. A self-hosted backend was considered, but Formspree was selected as the practical fit for this resume site.

  • Discussed JavaScript form submission without page reload.
  • Compared third-party form services with a self-hosted Linode endpoint.
  • Connected the form to https://formspree.io/f/xnjkqyyv.
  • Added required name, email, and message fields.
  • Added async fetch() submission, loading state, success/error messages, validation, and a honeypot.

Phase 8June 22-23, 2026

Final layout and content cleanup

The final pass focused on making the page easier to scan and removing duplicated contact information.

  • Enlarged section text and widened resume content areas.
  • Tuned divider widths, contact-card padding, and social-icon spacing.
  • Removed extra text from the Contact Me section so it contains only the heading and form.
  • Moved the contact card below the summary and increased its text size.
  • Renamed the sidebar label from "Explore" to "Navigate".

Phase 9June 23, 2026

Source copied back to the MacBook

After the remote Linode build work, the source repository was copied back into the local workspace so it could be maintained from the MacBook.

  • Copied /home/josh/resume-site/ from the Linode into ~/Documents/resume-site/.

Session context

Operational details from Codex

These details came from the actual build conversations and help explain how the site moved from idea to running production site.

Remote development loop

  1. SSH into josh@joshuacharrison.com.
  2. Edit the Hugo source in /home/josh/resume-site.
  3. Run hugo server --bind 127.0.0.1 on the Linode.
  4. Open an SSH tunnel from the MacBook to preview port 1313.
  5. Use the browser preview to check layout before publishing.

Production deployment loop

  1. Build production output with Hugo into /home/josh/resume-deploy.
  2. Keep generated files out of the source-editing path.
  3. Back up the current Caddy web root before major deploys.
  4. Use sudo rsync -a --delete to publish staged files into /usr/share/caddy.
  5. Verify HTTPS, Caddy, hostnames, CSS, and page layout after publishing.

Technical structure

How the site is organized

The project uses Hugo as a static-site generator. The source files define the site content, styling, and assets, while the generated output is staged separately before being copied into Caddy's live web root.

Configuration

hugo.toml defines the site URL, title, locale, description, and profile links.

  • baseURL = 'https://joshuacharrison.com/'
  • title = 'Joshua Harrison'
  • disableKinds = ['taxonomy', 'term']

Layout

layouts/index.html contains the one-page resume, navigation menu, contact form, footer, and small JavaScript behaviors.

Styling

assets/css/main.css defines the custom red and white visual identity, responsive layout, menu behavior, resume section spacing, and form states.

Static assets

static/img/ stores the profile image and social icons. Hugo copies these into the generated site output during the build.

Hosting

The production site runs on a Linode under joshuacharrison.com, with Caddy serving the static files and handling HTTPS.

Deployment path

Source edits happen in /home/josh/resume-site, production builds are staged in /home/josh/resume-deploy, and live files are copied into /usr/share/caddy.

Implementation workflow

Steps I took

The work followed a practical progression: establish hosting, prove the deployment path, choose the right design approach, turn the resume into web content, add interactivity, and bring the finished source back into the local repository.

Build steps

  1. Created and secured the Linode hosting environment.
  2. Configured Hugo, Caddy, DNS, and the production base URL.
  3. Built a minimal first homepage and staged it before deployment.
  4. Established the preview loop with Hugo server and SSH tunneling.
  5. Explored a resume theme, backed it up, then returned to the custom homepage.
  6. Converted the resume PDF into web-native sections.
  7. Added the portrait, profile links, contact card, and responsive refinements.
  8. Integrated Formspree with no-reload JavaScript submission and spam protection.
  9. Copied the final source from the Linode back to the local MacBook repository.

Key decisions

  • Used Hugo because a static resume site is fast, reliable, and easy to host.
  • Used Caddy because it made static hosting and HTTPS simpler on the Linode.
  • Reverted the resume theme because a custom layout better matched the intended personal brand.
  • Kept the site one page so recruiters and reviewers can scan the whole profile quickly.
  • Placed contact details near the top so the call to connect is visible immediately.
  • Chose Formspree over a custom backend because the site only needed a lightweight contact workflow.

Repository map

Files to reference

These are the primary files and directories that explain the project structure.

  • hugo.toml: site configuration, metadata, and profile links.
  • layouts/index.html: homepage content, resume sections, menu markup, contact form, and JavaScript.
  • assets/css/main.css: custom styling, responsive rules, and form/menu states.
  • static/img/: source image assets used by the site.
  • public/: generated static output produced by Hugo locally. It should be regenerated, not edited by hand.
  • /home/josh/resume-site: original Linode source path used during the remote build.
  • /home/josh/resume-deploy: staging directory used for production Hugo builds on the Linode.
  • /usr/share/caddy: live Caddy web root that serves the deployed site.
  • docs/build-process.html: this standalone manager-facing build-process document.
